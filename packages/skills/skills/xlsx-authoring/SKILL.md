# 电子表格建模惯例(通用)

一张专业的表的判据只有一条:**源数据变了,全表自动跟着对**。一切惯例都为此服务。

## 铁律:公式,不是死值
- 任何"算出来的数"(合计/占比/环比/单价×数量)一律 setFormula 引用源格——外部算好回填死值是头号反模式,源数据一改全表变错
- 常数参数(增长率/税率/汇率)放独立的假设单元格,公式引用它(=B5*(1+$B$6)),严禁把 1.05 这类常数埋进公式
- 必须硬编码的外部数据,在旁边注明来源与日期
- 除法/查找类公式用 IFERROR 包住,别让 #DIV/0!、#N/A 裸奔在交付物里

## 色码与数字格式惯例(金融建模通行)
- 蓝字=手输入参;黑字=公式;绿字=跨表引用——用户的表若已有这套色码,新增内容必须遵守
- 金额千分位 #,##0(表头注明单位量级);百分比默认一位小数;负数用括号;零显示为 "-"
- 年份按文本或 0 格式(千万别渲染成 2,024);估值倍数 0.0x
- 同列格式必须一致——一列里混着文本数字(如 "71")会让 SUM 静默漏算,发现即转数或明确提醒

## 查错闭环
- 公式错误五类速诊:#REF!(引用失效)/#DIV/0!/#VALUE!(类型错)/#NAME?(函数名)/#N/A(查无)
- 铺公式前先在 2-3 个格上用 read_range 验证取数正确,再铺满——行列 off-by-one 产出"貌似合理实则全错"的结果,是最难肉眼发现的错
- 投影期各列公式结构必须一致(有一列被手改过 → 拖期即断),规范化时顺手统一

## 尊重既有模板
- 改带格式的既有表:先看它的字体/色码/数字格式惯例并完全沿袭——模板既有规范优先于任何通用规范,"好心标准化"是破坏交付物
- 大表操作前 freeze 表头、开 filter;录入区加 dataValidation 防后续脏数据

## 反例
- 在 plan 里说"合计是 1180"然后 setValue 写死(要 =SUM 并让表自己算)
- 把 %/¥/千分位敲进 setValue 的值里(用 setNumberFormat,别把数字变文本)
- 对用户的美式财务模板强加自己的配色
