# 两个杀手实验 · 可执行方案

> 目的:在写任何 Agent 代码之前,用 1 周、近乎 $0,先证伪两个最可能让整个产品崩掉的假设。
> 原则:不建产品、只用免费工具、拿真实文件、定死判定线。任一证伪 → 立即重定方向。

---

## 实验一:保真度实验(能不能打开真实文件而不毁容)

### 它要回答的生死问题
"用户上传自己的真实 .xlsx/.docx,光是被引擎读进来再写回去(还没让 Agent 动),会不会就把图表/透视/条件格式/下拉/样式毁掉?" 这是头号 kill-shot。

### 测什么:三种"写回策略"对比
对**同一批文件**做"导入→导出、不做任何编辑"的往返,比三条路:

| 策略 | 怎么做 | 代表的产品路线 |
|---|---|---|
| **A. LibreOffice 往返** | `soffice` headless 把文件读进 LO 模型再存回同格式 | 自托管免费高保真引擎(主推) |
| **B. OnlyOffice 往返**(可选) | Docker 跑 DocumentServer,用其转换接口 / `x2t` 往返 | Word 线的候选引擎 |
| **C. 外科手术补丁** | 把 OOXML 当 zip,只改目标部件的 XML、其余字节原样,重新打包 | 真正的"只重写被编辑子树、其余透传"策略 |
| D. SheetJS 往返(对照) | `node` 用 `xlsx` 库读再写 | 纯前端有损路线(预期会毁容,作反面基准) |

> 关键洞察:如果 **C(外科补丁)** 能保住一切,那么"用 Univer 做编辑 UX、但写回走外科补丁"就能既好用又保真——这是最值得验证的架构假设。

### 拿哪些文件:20 个 .xlsx + 20 个 .docx(覆盖矩阵)
不要随便抓,要**故意覆盖最容易崩的特性**,这样能学到"哪些特性安全、哪些危险":

- **Excel 必含**:图表、数据透视表、条件格式、数据验证(下拉)、切片器、合并单元格、命名样式/主题、批注、VBA 宏、动态数组溢出、结构化引用(表)。
- **Word 必含**:修订(track changes)、分栏、题注/交叉引用、目录、页眉页脚、嵌入图表/对象、复杂表格、样式、公式(equation)、图文混排。
- **来源**(免费):① 你自己/同事的真实财报·统计表·论文·教案(最有代表性);② 国家统计局/财政公开的 .xlsx;③ Microsoft 官方模板库;④ GitHub 上带这些特性的真实样本。
- 做一张「文件 × 特性」覆盖表,确保 20+20 文件合起来把上面每个特性都至少覆盖 2 次。

### 怎么量丢失(三层,客观+人工)
1. **结构差异(脚本,客观)**:.xlsx/.docx 本质是 zip。往返前后各解压,比对关键部件是否还在/数量是否变:
   - Excel:`xl/charts/`、`xl/pivotTables/`、sheet XML 里的 `<conditionalFormatting>`/`<dataValidations>`、`xl/drawings/`、`xl/vbaProject.bin`、`xl/styles.xml`。
   - Word:`word/charts/`、`word/embeddings/`、`word/header*.xml`、`word/styles.xml`、`document.xml` 里的修订标记 `<w:ins>/<w:del>`。
   - 工具:Python `zipfile` + `openpyxl`/`python-docx` 清点;一个脚本跑完 40 个文件。
2. **视觉差异(脚本)**:`soffice --convert-to pdf` 把往返前后各转 PDF → `pdftoppm` 转 PNG → 逐页像素/感知差异(Pillow / ImageMagick `compare`)。抓版式漂移。
3. **人工对照(地面真值)**:在**真 Excel/Word**里打开往返前后,按清单逐项打钩:图表在否、透视能否刷新、下拉还在否、条件格式在否、版式是否一致、公式数值是否变。40 个文件人工抽检约半天。

### 判定线(动工前就定死)
- 给每个文件定"是否毁容":出现以下任一即毁容 = {图表丢、透视丢、条件格式丢、下拉丢、明显版式漂移、公式数值变}。
- 指标:**每条策略的"干净往返率" = 未毁容文件数 / 总数**。
- 决策规则:
  - 若 **C 外科补丁干净率 ≥ 95%**(几乎必然,因为不动其余字节)→ ✅ 写回走外科补丁,保真问题解决,继续。
  - 若 C 不可行,但 **A 或 B(LibreOffice/OnlyOffice)干净率 ≥ 85%** → ✅ 自托管引擎兜底成立,继续。
  - 若**最好的免费策略干净率 < 70%** → 🔴 "打开真实文件"前提在免费方案下也不成立 → 必须重定:① 只支持往返安全的文件子集;② 或转"新建文档"形态;③ 或承认需更重方案。
- 附产出:**一张"特性 × 策略 × 存活"表**——直接定义 MVP 第一版"支持改哪些、暂不碰哪些"。

### 成本与产出
免费(LibreOffice/Docker/脚本全免费)。产出:干净往返率表 + 推荐写回策略 + MVP 支持范围。约 2 天。

## 实验二:意图覆盖 + 交互实验(原语成不成立)

### 它要回答的两个生死问题
A. 真实用户的指令,能不能被"结构化 ChangeSet 指令 + 技能"表达?还是长尾宽到约束化 Agent 根本做不到?(可靠性 vs 能力之争)
B. 目标用户到底**会不会主动"圈选区域"**,还是更想直接打字/点按钮?(核心原语是否成立)

### Part A — 意图覆盖(不需要产品)
1. **收 100 条真实自然语言指令**:
   - 找 10–20 个真实目标用户(学生/老师/办公同事),让每人**对自己的一份真实文件**写下 5–10 条"我想让 AI 帮我做的事"。→ 得到 ~100 条**有真实文件背景**的指令。
   - 补充挖矿:知乎/百度知道/r/excel/superuser 上"Excel/Word 怎么做 X"的高频问题。
2. **逐条分类**到三档:
   - **核心 DSL 可表达**:setValue/setFormula/setStyle/insertRows/sort/fill/… 的组合(Excel);insertText/replaceRange/setStyle/… (Word)。
   - **DSL + 一个技能可表达**:需要打包的高层操作(做透视、条件格式高亮异常、按季度分组、按某模板排版)。
   - **够不着**:当前建模无法表达。
3. **产出**:① 三档占比;② **从真实指令反推出来的 DSL 算子清单**(不是拍脑袋,是数据驱动);③ **第一批必须做的技能清单**。

### Part B — 交互验证(纸面/可点击原型,无需真 Agent)
1. 做一个**可点击 Figma 原型 / 甚至纸面**:左 Office 文档 + 右聊天框。给 15–20 个目标用户一个任务("把这块数据做成图、超标标红"),用 think-aloud 观察:
   - 他们是**本能地在文档上圈/选那块**,还是直接去聊天框打字,还是找工具栏?
   - 是否有人说"为什么不能直接告诉它"?
2. 顺带测**红笔批注语言(创新①)**:把圈/箭头/删除线摆出来,看他们是否**不教就懂**这些=命令。
3. 全程零后端,Wizard-of-Oz 即可(你在后台假装 Agent)。

### 判定线
- **覆盖率**:若"核心 DSL + 合理技能集"能覆盖 **≥ 65%** 真实指令 → ✅ 约束化 Agent 路线可行;若大量"够不着" → 🔴 收窄场景或换思路。
- **交互**:若 **≥ 40–50%** 用户**自发圈选**且觉得自然 → ✅ 原语成立,圈选优先;若几乎无人圈选、都想打字 → 🟡 把圈选降级为可选,主交互回到"选区/对象 + 打字"。

### 成本与产出
≈$0(招的人可以是同学/同事;分类可借 LLM 提速,几毛钱 API)。产出:意图三档占比 + 数据驱动的 DSL 算子清单 + 首批技能清单 + 交互结论。约 2–3 天。

## 公开保真 benchmark(项目一等资产)

把"20 xlsx + 20 docx × 特性 × 写回策略 × 存活率"做成**公开、可复现、可引用的基准**(repo 里 `benchmark/` + 一页结果表)。价值:
- **护城河**:系统性公开 Office 往返保真——没人敢做,做了即成行业参照。
- **信誉与传播**:开源项目靠它立专业度、做 SEO/口碑。
- **透明诚实**:契合全开源精神,也持续约束自己不毁文件。

覆盖维度:**引擎**(LibreOffice / 外科补丁 / SheetJS / python-docx …)× **特性**(图表/透视/条件格式/数据验证/宏/页眉页脚/脚注/目录/嵌入对象 …)× **存活层级**(字节级 / 视觉 / 功能)。每次发版回归跑,结果进 README 徽章。
