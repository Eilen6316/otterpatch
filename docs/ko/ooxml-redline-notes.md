# OOXML 수정 추적(Redline) 시맨틱 노트 (adapter-word 기여자용)

쓰기 되돌림(write-back) 계층은 **진짜 Word 수정 추적(revision)** 을 생성해야 하며, 이 시맨틱 세부 사항들이 "수정 사항 수락 후 문서가 깨끗한지"를 결정한다. 현재 adapter-word는
일부를 이미 커버하고 있으며, 미커버 항목은 backlog로 표시했다. 출처: OOXML 규격 및 주요 구현체에 대한 관찰이며, 본문은 이 프로젝트의 자체 작성물이다.

## 이미 커버됨(테스트 있음)
- 삽입 = `<w:ins>`로 run을 감싼다; 삭제 = `<w:del>`로 run을 감싸며, 그 안의 텍스트 노드는 반드시 `<w:delText>`로 이름을 바꿔야 한다
- 수정 최소화: 변경된 단어만 del/ins 쌍으로 잘라내고, 앞뒤의 변경되지 않은 텍스트는 원본 run 바이트를 그대로 유지한다
- 문자 서식 수정 `<w:rPr>+<w:rPrChange>`, 단락 서식 수정 `<w:pPr>+<w:pPrChange>`
- 수정 run은 반드시 원본 `<w:rPr>`을 복사해야 한다. 그렇지 않으면 수정 수락 후 굵게/글자 크기가 사라진다
- 페이지 수준 sectPr 패치(cols/pgMar/pgSz), OOXML 요소 순서에 따라 삽입

## Backlog(미커버, PR 환영)
- **단락 전체 삭제 시 단락 기호 표시**: 삭제 내용 run 외에, 해당 단락의 `<w:pPr><w:rPr>` 안에 빈 `<w:del/>`을 넣어
  단락 기호 자체가 삭제되었음을 표시해야 한다 — 이것이 없으면 수정 수락 후 빈 단락/빈 목록 항목이 남는다. 현재 replaceText는 단락 전체를 비우는 경우를 처리하지 않는다.
- **중첩 거부(veto) 시맨틱**: 타인의 삽입을 거부 = 상대방의 `<w:ins>` 안에 자신의 `<w:del>`을 중첩; 타인의 삭제를 복원 =
  상대방의 `<w:del>`을 유지하고 그 뒤에 자신의 `<w:ins>`를 추가해 같은 텍스트를 다시 쓴다. 다중 저자 협업 시나리오에서 필요하다.
- **`xml:space="preserve"`**: 선행/후행 공백이 있는 `<w:t>`를 생성할 때 반드시 붙여야 하며, 그렇지 않으면 공백이 조용히 사라진다.
  현재 생성 경로에서는 체계적으로 검증하지 않는다.
- **`<w:pPr>` 자식 요소 순서 schema**: pStyle → numPr → spacing → ind → jc → rPr(맨 뒤);
  pPrChange 주입 시 원본 단락에 pPr이 없다면 새로 만드는 것도 이 순서를 지켜야 한다.
- **주석(comments)**: 앵커 `commentRangeStart/End`는 run의 형제 노드(w:p의 직접 자식)이며,
  run 안에 넣을 수 없다; 참조 표식은 독립된 run이다. 향후 "Agent가 본문을 고치지 않고 주석만 남기는" 모드의 기반이 된다.
- **단위 체계**: DXA(1440 = 1인치)는 페이지/들여쓰기/표에 사용; EMU(914400 = 1인치)는 이미지에 사용.
  sectPr 패치는 이미 DXA를 사용 중이며, 향후 이미지 삽입에는 EMU + 4단계 등록(media/ + rels + Content_Types + w:drawing)이 필요하다.

## 검증 아이디어
- 쓰기 되돌림 후의 docx는 다음을 통과해야 한다: 압축 해제 → 모든 수정 수락(LibreOffice headless로 자동화 가능) → "직접 수정한 후의
  텍스트"와 비교해 일치 + 잔여 빈 단락 없음; 이는 "열리기만 하면 된다"보다 강한 정확성 판정 기준이므로 CI에 넣을 가치가 있다.
