# 에이전트 루프 (The Agent Loop)

모델이 할 수 있는 모든 것과, 우리가 이를 어떻게 정직하게 유지하는지.

## 라우팅 (세 개의 출구, 하나의 루프)

시스템 프롬프트(`ROUTING_PREAMBLE`, `packages/agent/src/prompts/agent-loop.ts`)가 계약을
고정합니다: 모델은 모든 턴을 반드시 정확히 하나의 툴 호출로 끝내야 합니다 —

- `answer_user` — 질문/컨설팅; 문서에는 절대 손대지 않음
- `propose_changeset` — **유일한** 변경(mutation) 출구; 먼저 계획을 세우고, 편집은 반영 전에 리뷰됨
- `ask_user` — 의도가 진짜로 모호하고 추측 비용이 클 때 사용하는 가이드형 명확화 테이블
  (질문당 2–4개 선택지)

그 사이에 모델은 최대 `STEP_LIMIT = 8` 루프 스텝 동안 **읽기 툴**을 호출할 수 있습니다. 두 모델
채널(`anthropic.ts`, `openai-compat.ts`) 모두 `sheet-tools.ts` / `doc-tools.ts`에 있는 공유
프로바이더 비종속(provider-agnostic) 툴 정의 위에서 동일한 루프를 구현합니다.

## 읽기 툴 (행동하기 전에 인지하기)

| 포맷 | 툴 | 목적 |
|---|---|---|
| Excel | `read_range` | 임의의 A1 범위에 대한 정확한 셀 값 (샘플로부터 절대 추측하지 않음) |
| Excel | `aggregate` | `groupBy` / `where`를 지원하는 열 집계 — 피벗, 합계, 이상치 통계 |
| Word | `read_blocks` | 문단 범위의 전체 텍스트 (프롬프트 컨텍스트는 긴 문단을 잘라내므로 — 인용문은 실제 텍스트에서 가져와야 함) |
| Word | `find_text` | 블록 번호가 포함된 모든 출현 위치 — 인용문 유일성 검사 |
| Word | `get_outline` | 제목(heading) 트리 + 레벨 건너뛰기 진단 |
| Word | `get_style_usage` | 스타일/글꼴/크기/정렬 분포 — 타이포그래피 감사(audit)의 원재료 |
| 공통 | `load_skill` | 도메인 플레이북의 전체 지침 불러오기 ([skills.md](./skills.md) 참조) |

스냅샷은 요청에 함께 실려 오며(`ProposeRequest.sheet` / `.doc`) 툴에게만 보입니다 — 프롬프트에
붙여 넣어지지 않습니다.

## 섀도 검증: 제안 → 관찰 → 수리 (propose → observe → repair)

모든 `propose_changeset`은 diff가 되기 전에 검증됩니다 (검증기 레지스트리는
`packages/runtime/src/runtime.ts`의 `registerVerifier(format, make)`):

- **Excel** (`buildGridVerifier`) — 재계산 + 범위 경계 + 중복 히트 검사
- **Word** (`buildDocVerifier`) — 모든 인용문은 원본 텍스트에 존재해야 **하고 유일해야** 함; 빈
  편집과 중복 히트는 플래그 처리됨
- **drawio** (`buildDrawioVerifier`) — 토폴로지 무결성: `update/delete/move` 대상은 반드시 존재해야
  함; 새 엣지는 기존 노드 또는 같은 제안(proposal) 내의 노드에 연결되어야 함 (매달린 엣지 금지);
  중복 id 금지

실패는 같은 턴 안에서 구조화된 리포트로 모델에게 반환되며, 모델은 최대 `maxRepairs = 2`회까지
수리할 수 있습니다.

**최종 자기 점검** (`withFinalSelfCheck`): *큰* 체인지셋(편집 5개 이상)이 구조 검증을 통과하면,
모델은 정확히 한 번 "자신의 작업 전체를 리뷰하는" 라운드를 갖습니다 — 완전성, 충돌, 더 나은
접근 — 그 후 다시 제출합니다 (만족스러우면 변경 없이 그대로). 작은 체인지셋은 이 단계를
건너뜁니다.

## 프롬프트 캐싱

Anthropic 채널은 시스템 프롬프트를 **안정 프리픽스**(라우팅 + 방언(dialect) + 스킬 — 턴 간에
동일)와 **가변 테일**(이번 턴의 문서 스냅샷)로 분리하며, 각각 `cache_control` 브레이크포인트를
가집니다. 결과: 8스텝 루프의 모든 스텝이 시스템 프롬프트 전체에 대해 캐시를 히트하고, 턴을
넘어서도 안정 프리픽스는 여전히 캐시를 히트합니다.

## 배치 처리 (직렬, 절대 병렬 아님)

긴 출력은 배치로 분할됩니다: 계획은 "처음 N개 항목"을 선언하고, 수락된 후 사용자는
**继续下一批**(다음 배치 계속)를 클릭할 수 있습니다 — 또는 **⚡自动续批**(자동 연속 배치,
옵트인, 설정 유지)를 켤 수 있는데, 이는 각 수락 후 "下一批"(다음 배치)를 자동으로 전송하며,
연속 자동 배치는 5회로 제한됩니다. 각 배치는 *현재* 문서를 기준으로 앵커링된 완전한 제안 →
검증 → 리뷰 라운드입니다.

배치에 병렬 서브 에이전트를 쓰지 않는 이유는? 모든 앵커(인용문 / A1 참조)는 하나의 문서
리비전에 대해 해석되는데, 배치 A가 반영되는 순간 배치 B의 앵커는 낡은(stale) 것이 됩니다 —
조용한 무시(no-op) 또는 잘못된 위치에 반영된 편집으로 이어집니다. 병렬성은 **읽기**(진단
팬아웃)에는 안전하지만, 쓰기는 직렬로 앵커링된 단일 체인지셋으로 수렴해야 합니다. 언젠가 서브
에이전트가 도입된다면 설계는 다음과 같습니다: 병렬 리더 → 단일 라이터 → 단일 체인지셋 → 단일
리뷰.

## 모델이 보는 히스토리와 상태

`buildHistory`는 과거의 각 턴을 **순(net) 결과**를 포함해 한 줄로 투영합니다 — "사용자가 N개
항목을 수락함" / "사용자가 이것들을 되돌림" — 그래서 모델은 이미 반영된 변경을 다시 제안하거나
되돌려진 변경 위에 쌓아 올리지 않습니다. 승인 상태는 컨텍스트 트리밍에도 살아남습니다 (잘려
나간 턴은 상태 요약을 남깁니다).
