# 스킬 (Skills)

`packages/skills`는 지식 허브입니다. 에이전트가 *잘 수행하는 방법을 아는 것*을 코어 프롬프트 밖에 보관하여, 모든 요청을 비대하게 만들지 않으면서 지식을 확장할 수 있게 합니다.

## 두 가지 종류의 내장 스킬

| 종류 | 예시 | 내용 | 주입 방식 |
|---|---|---|---|
| **역량 카드 (Capability cards)** | `xlsx`, `docx`, `pptx`, `pdf`, `drawio` | "이 형식이 지원하는 것"을 한 줄로 요약 | L0: 시스템 프롬프트에 이름 + 설명 |
| **플레이북 (Playbooks)** | `docx-gongwen`, `xlsx-financial`, `chart-selection` | 체크리스트 + changeset 관용구 + 안티패턴 | L0 카드에 【플레이북 있음】 태그 표시; 전체 텍스트는 `load_skill`을 통해 필요 시 제공 |

내장 플레이북은 `packages/skills/skills/<name>/SKILL.md` 아래에 **실제 SKILL.md 파일**로 존재합니다
(Anthropic Agent Skills 디렉터리 규약 — 스킬당 폴더 하나, L0용 YAML frontmatter,
L1용 마크다운 본문). `playbooks.ts`는 단순한 로더일 뿐이며, 마크다운만 편집하면 되고 코드 변경은 필요 없습니다:

- **`docx-gongwen`** — GB/T 9704 공문(公文, 중국 공식 행정문서) 레이아웃: 제목/본문 글자 크기 체계(제목은 二号 小标宋(샤오뱌오쑹, 공문 제목용 서체), 본문은 三号 仿宋(팡쑹체)), 一、/(一)/1./(1) 제목 번호 계층 구조, 전각 문장부호,
  그리고 이를 적용하기 위한 changeset 관용구(`all=true` 일괄 적용 전에 실제 제목에는 `block`을 먼저 사용).
- **`xlsx-financial`** — 대사(reconciliation) 검증(합계는 반드시 수식으로, 하드코딩 금지), 통화/백분율
  숫자 서식, 이상치 탐지, 그리고 절대 원칙: 수치 결론을 쓰기 전에 반드시 `read_range`/`aggregate`로
  검증할 것; 원본 데이터를 절대 덮어쓰지 말 것.
- **`chart-selection`** — 질문 → 차트 유형으로 이어지는 의사결정 트리와 전문성 최저 기준 규칙
  (막대그래프는 0 기준선, 색상 6개 이하, 의미 있는 정렬 순서, 결론형 제목).

## 점진적 공개 (Progressive disclosure)

1. `SkillLibrary.match(intent, format)`가 카드에 점수를 매깁니다: 형식 일치 +3, 키워드 일치당 +1,
   그리고 **키워드가 실제로 일치한 경우에만** 플레이북에 +0.5의 동점 처리 가산점(일반적인 의도에서는
   여전히 일반 역량 카드가 먼저 순위에 오릅니다).
2. `render()`가 상위 카드(L0)를 시스템 프롬프트에 주입합니다. 플레이북 카드에는 명시적 지시가
   포함됩니다: *관련이 있다면 행동하기 전에 `load_skill`로 로드하라*.
3. `load_skill` 도구(`Agent.withSkillTools`가 `RespondOptions.extraTools`를 통해 연결)는
   플레이북의 전체 마크다운을 도구 결과로 반환합니다 — 지식은 필요할 때만 도착합니다.

## 외부 스킬

업계 또는 팀 특화 내용은 내장 스킬에 포함하지 않습니다. 호스트는 런타임에 표준
`SKILL.md`(Anthropic Agent Skills 호환 — YAML frontmatter + 마크다운 본문)를 설치할 수 있습니다:

```ts
library.install(skillMdText, 'file:./skills/my-company-report.md');
```

파싱된 카드는 즉시 매칭/렌더링/`load_skill`에 합류합니다. L2(실행 가능한 스크립트)는
의도적으로 활성화하지 않았습니다 — 텍스트 플레이북만으로도 샌드박스 위험 없이 대부분의 가치를 제공합니다.

## 좋은 플레이북 작성법

- **모델이 행동 전에 실행하는 체크리스트**로 시작하세요 — 처방보다 진단이 낫습니다.
- **changeset 관용구**를 포함하세요: 어떤 op를 어떤 순서로 사용하는지(예: "먼저 `block`으로 실제
  제목을 설정한 다음, 본문 기준선에 `all=true`를 적용하라 — 그렇지 않으면 일괄 적용이 제목을 뭉개버린다").
- **안티패턴**을 포함하세요("수동 굵게+크기 조정으로 가짜 제목을 만들지 말 것").
- 약 50줄 이내로 유지하세요. 라이브 컨텍스트에 로드되는 문서이므로 밀도가 승부를 가릅니다.
