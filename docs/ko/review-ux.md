# 리뷰 경험 (The Review Experience)

에이전트의 모든 제안은 사이드 패널만이 아니라 **작업 공간 안에서, 그 자리에서(in-place)** 리뷰됩니다.

## Word: 인라인 변경 내용 추적

제안은 세 가지 채널의 인라인 마크로 표시됩니다(Word의 변경 내용 추적 / Google Docs의
제안 모드와 유사):

- **삽입(Insertions)** — 초록색, 밑줄 (`ins.rd-ins`)
- **삭제(Deletions)** — 빨간색 취소선, 시각적으로 뒤로 물러난 표현 (`del.rd-del`)
- **서식 변경(Format changes)** — 점선 밑줄 + 작은 글리프 칩 (`B`/`I`/`U`/`A±`/`¶`…)

그 주변에는:

- **4단계 보기 토글**(페이지 위에 떠 있음): 原文(원문) / 修订(수정 표시) / 清样(교정쇄) / 改后(수정 후) —
  원본, 전체 마크업, 변경 표시줄만 있는 클린 보기, 최종본. 변경 개수 카운터와
  ‹ › 단계 이동이 있는 슬라이딩 썸(segmented control) 방식.
- **변경 항목별 호버 카드** — 유형, 이전 → 이후, ✓ 수락 / ✕ 거부를 읽고 있는 바로 그 자리에서 제공.
  키보드: Tab으로 변경 항목 이동, Enter/Space로 카드 열기.
- **거터 변경 표시줄(gutter change bars)** — 변경이 포함된 모든 블록에 표시; 레일 ↔ 인라인 호버가 양방향으로 연동.
- **문서 수준 칩** — `all=true`(문서 전체 서식) 및 페이지 수준 변경(단 나누기 /
  여백 / 방향)은 인라인 앵커가 없으므로 토글 옆에 자체 ✓/✕가 있는 칩으로 표시됩니다.
  原文(원문) 보기는 이를 실제로 되돌려(글꼴, 단 수, 페이지 상태) 진짜 전/후 비교가
  가능합니다.

### 수락 시 평탄화(Flatten-on-accept) — 아키텍처의 핵심

**수락은 표면적인 것이 아니라 물리적입니다.** 수락 시 삭제 부분은 DOM에서 제거되고,
삽입 부분은 일반(스타일 적용된) 콘텐츠로 언랩되며, 모든 수정(revision) 속성이 제거됩니다.
래퍼는 *이번 라운드의* 실행 취소를 유지하는 보이지 않는 `data-undo` span으로 격하되며,
다음 제안이 도착하면 정리(sweep)됩니다.

이것이 중요한 이유 — 다운스트림의 모든 것이 **구조적으로(by construction)** 깨끗하게 유지됩니다:

- 에이전트 컨텍스트(`getText`/`getContext`)는 *클린 프로젝션*(삭제 부분 제외)을 읽으므로,
  다음 턴의 인용문이 이전/새 텍스트에 걸칠 수 없습니다 — 누적되는 손상 루프가 없습니다.
- 단어 수, 찾기 및 바꾸기, 인쇄, 복사가 모두 실제 문서만 봅니다.
- 리뷰 도중 새로고침해도 안전합니다: 수락 기록(`changeSetId::editId`)이 유지되고, `applyEdit`은
  멱등(idempotent)이며, 메모리 내 undo 맵이 사라진 경우 거부는 DOM 수준 복원으로 폴백합니다.

### 배치 UX

계획이 배치를 선언하면("先做第一批…" — 먼저 첫 번째 배치를 수행), 수락된 턴에 **继续下一批 ›**(다음 배치 계속)와
**⚡自动续批**(자동 배치 이어가기) 옵트인 토글(설정 유지됨)이 표시됩니다. 자동 계속은 각 수락 후
"下一批"(다음 배치)를 전송합니다 — 순차적으로, 각 배치는 다시 앵커링되고 다시 리뷰되며 — 연속 최대 5개
배치로 제한됩니다.

## Excel: 이전 상태 리플레이

제안 시점에 데스크톱은 변경 대상이 된 각 셀의 **완전한 이전 상태** — 값 *및
수식*, 채우기, 글꼴 색, 굵게 — 를 캡처합니다. 거부(또는 原文(원문) 보기)는 해당 작업이
건드린 차원만 정확히 리플레이합니다: 값 편집을 거부해도 같은 셀의 스타일 편집이 망가지지 않고,
수식은 수식으로 복원되며, 사용자의 채우기는 살아남습니다. 原文/改后(원문/수정 후) 빠른 토글은 항목별
결정을 존중합니다(거부된 편집이 보기를 전환한다고 되살아나지 않습니다).

## 레일 (두 포맷 공통)

항상 표시되는 **git 스타일 통합 diff**: 빨간 − / 초록 + 줄과 `~` 서식 줄이 있는
`@@ ref label` 헝크(hunk), 항목별 수락/거부, 진행률 표시줄, 실행 취소가 있는 "已采纳 · N 处"(수락됨 · N곳).
헝크에 호버하면 대응하는 인라인 변경이 강조되고, 그 반대도 마찬가지입니다.

## 텔레메트리

항목별 결정마다 포맷 × 변경 유형(`text` / `style` / `value` / `structure` / `object`)을 키로 하는
`localStorage['oa.telemetry']` 카운터가 증가합니다. 콘솔에서 `__otterTelemetry()`로 읽을 수 있습니다.
카테고리별 수락률은 "에이전트가 정말 전문가인가?"에 대한 실측(ground-truth) 지표입니다 —
점수가 가장 낮은 카테고리가 다음 플레이북/프롬프트 개선 대상입니다.
