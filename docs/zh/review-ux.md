# 审阅体验

每一条 Agent 提案都是**就地、在工作区内**审阅的——而不只是在侧边面板里。

## Word：行内修订标记

提案以三通道行内标记落地（类似 Word 的修订 / Google Docs 的建议模式）：

- **插入** —— 绿色、带下划线（`ins.rd-ins`）
- **删除** —— 红色删除线，视觉上后退（`del.rd-del`）
- **格式变更** —— 点状下划线 + 一枚小字形标签（`B`/`I`/`U`/`A±`/`¶`……）

围绕这些标记还有：

- **四态视图切换**（悬浮于页面之上）：原文 / 修订 / 清样 / 改后 —— 即原始文档、完整标记、
  仅保留改动条的清爽视图、最终结果。滑块式分段控件，附带改动计数器和 ‹ › 逐条导航。
- **逐条改动悬浮卡片** —— 显示类型、旧值 → 新值、✓ 接受 / ✕ 拒绝，就在你阅读的位置。
  键盘操作：Tab 跳到某处改动，Enter/Space 打开卡片。
- **边栏改动条**：任何包含改动的块都会显示；侧栏 ↔ 行内悬停双向联动高亮。
- **文档级标签（chip）** —— `all=true`（全文格式）和页面级改动（分栏 / 页边距 / 纸张方向）
  没有行内锚点，因此以标签形式出现在切换器旁，带各自的 ✓/✕；原文视图会真正回退它们
  （字体、栏数、页面状态），实现真实的前后对比。

### 接受即压平（Flatten-on-accept，架构核心）

**接受是物理性的，不是装饰性的。** 接受时，删除内容会从 DOM 中移除，插入内容被解包为普通
（带样式的）内容，所有修订属性都被剥离。包裹元素退化为一个不可见的 `data-undo` span，
用于保证*本轮*的撤销仍然可用；下一条提案到来时会被清扫掉。

为什么这很重要——下游的一切都**在构造上**保持干净：

- Agent 上下文（`getText`/`getContext`）读取的是*干净投影*（不含删除内容），因此下一轮的
  引用不会横跨新旧文本——不会出现累积腐化循环。
- 字数统计、查找替换、打印、复制看到的都只是真实文档。
- 审阅中途重载是安全的：接受记账（`changeSetId::editId`）会持久化；`applyEdit` 是幂等的；
  当内存中的撤销映射丢失时，拒绝会回落到 DOM 级还原。

### 分批体验

如果计划声明了分批（"先做第一批……"），已接受的轮次会显示**继续下一批 ›**，外加一个
**⚡自动续批**可选开关（持久化）。自动续批会在每次接受后发送"下一批"——串行执行，
每一批都重新锚定、重新审阅——最多连续 5 批。

## Excel：前置状态回放

在提案时刻，桌面端会捕获每个被触碰单元格的**完整前置状态**——值*和公式*、填充色、字体颜色、
加粗。拒绝（或原文视图）会精确回放该操作触碰过的那些维度：拒绝一处值编辑不会破坏同一
单元格上的样式编辑，公式恢复为公式，用户填充色得以保留。原文/改后快速切换会尊重逐项决策
（已拒绝的编辑不会在切换视图时复活）。

## 侧栏（两种格式通用）

一个始终可见的 **git 风格统一 diff**：`@@ ref label` 区块，红色 − / 绿色 + 行以及 `~` 格式行，
逐项接受/拒绝，进度条，"已采纳 · N 处"并可撤销。悬停某个区块会点亮对应的行内改动，
反之亦然。

## 遥测

每一次逐项决策都会递增 `localStorage['oa.telemetry']` 中以格式 × 改动类型
（`text` / `style` / `value` / `structure` / `object`）为键的计数器。在控制台通过
`__otterTelemetry()` 读取。每个类别的接受率就是"Agent 到底专不专业？"的真值指标——
得分最低的类别就是下一个 playbook/prompt 的优化目标。
