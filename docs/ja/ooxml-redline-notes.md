# OOXML 変更履歴セマンティクスに関するノート(adapter-word コントリビューター向け)

書き戻し層は**本物の Word 変更履歴(修訂)**を生成しなければならず、これらのセマンティクス上の細部が「変更を承諾した後にドキュメントがクリーンであるか」を左右する。現時点で adapter-word
は一部をカバー済み。未カバー項目は backlog と明記する。出典:OOXML 仕様および主要実装の観察に基づき、本文は本プロジェクトのオリジナルである。

## カバー済み(テストあり)
- 挿入 = `<w:ins>` で run を包む。削除 = `<w:del>` で run を包み、かつその中のテキストノードは `<w:delText>` に改名しなければならない
- 変更履歴の最小化:変化した語句だけを del/ins のペアに切り出し、前後の未変更テキストは元の run のバイト列をそのまま保持する
- 文字書式の変更履歴 `<w:rPr>+<w:rPrChange>`、段落書式の変更履歴 `<w:pPr>+<w:pPrChange>`
- 変更履歴の run には元の `<w:rPr>` を複製しなければならない。さもないと変更承諾後に太字やフォントサイズが失われる
- ページレベルの sectPr パッチ(cols/pgMar/pgSz)、OOXML の要素順序に従って挿入する

## Backlog(未カバー、PR 歓迎)
- **段落全体削除時の段落記号マーク**:削除内容の run に加えて、当該段落の `<w:pPr><w:rPr>` 内に空の `<w:del/>`
  を置いて段落記号自体が削除されたことを示す必要がある——これが欠けると変更承諾後に空段落や空のリスト項目が残留する。現在の replaceText は段落全体を空に置換するケースを未処理。
- **ネストされた却下(否決)セマンティクス**:他者の挿入を却下する = 相手の `<w:ins>` の内側に自分の `<w:del>` をネストする。他者の削除を復元する =
  相手の `<w:del>` を保持し、その直後に自分の `<w:ins>` を追加して同一テキストを書き直す。複数著者による共同編集シナリオで必要となる。
- **`xml:space="preserve"`**:先頭または末尾に空白を含む `<w:t>` を生成する際は必ず付与しなければならない。さもないと空白が黙って失われる。
  現在の生成パスでは体系的な検証を行っていない。
- **`<w:pPr>` 子要素の順序スキーマ**:pStyle → numPr → spacing → ind → jc → rPr(最後尾)。
  pPrChange を注入する際、元の段落に pPr がない場合、新規作成するものはこの順序を守らなければならない。
- **コメント(comments)**:アンカーの `commentRangeStart/End` は run の兄弟ノード(w:p の直接の子ノード)であり、
  run の中に押し込んではならない。参照マークは独立した run である。将来の「Agent が本文を変えずにコメントを残す」モードの基礎となる。
- **単位体系**:DXA(1440 = 1 インチ)はページ/インデント/表に使用。EMU(914400 = 1 インチ)は画像に使用。
  sectPr パッチはすでに DXA を使用済み。将来の画像挿入には EMU に加えて 4 ステップの登録(media/ + rels + Content_Types + w:drawing)が必要。

## 検証の考え方
- 書き戻し後の docx は次を通過すべき:展開(unzip)→ すべての変更履歴を承諾(LibreOffice headless で自動化可能)→「直接編集後の
  テキスト」と比較して一致 + 残留空段落なし。これは「開ける」よりも強い正しさの判定基準であり、CI に組み込む価値がある。
