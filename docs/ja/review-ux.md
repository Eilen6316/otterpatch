# レビュー体験

エージェントのすべての提案は、サイドパネルだけでなく、**ワークスペース内でその場で（in-place）** レビューされます。

## Word: インライン変更履歴（tracked changes）

提案は、3 チャネルのインラインマークとして反映されます（Word の変更履歴 / Google Docs の提案モードに相当）:

- **挿入** — 緑色・下線付き（`ins.rd-ins`）
- **削除** — 赤色の取り消し線、視覚的に後退表示（`del.rd-del`）
- **書式変更** — 点線の下線 + 小さなグリフチップ（`B`/`I`/`U`/`A±`/`¶`…）

その周辺には:

- **4 状態のビュートグル**（ページ上にフローティング表示）: 原文 / 修訂 / 清様 / 改後 — すなわち「オリジナル」「フルマークアップ」「変更バー付きクリーン表示」「最終版」。スライド式サムのセグメントコントロールで、変更カウンターと ‹ › のステップナビゲーション付き。
- **変更ごとのホバーカード** — 種別、旧 → 新、✓ 承認 / ✕ 却下を、読んでいるその場所に表示。キーボード操作: Tab で変更へ移動、Enter/Space でカードを開く。
- **ガター変更バー** — 変更を含むすべてのブロックに表示。レール ↔ インラインのホバー連動は双方向。
- **ドキュメントレベルのチップ** — `all=true`（文書全体の書式）およびページレベルの変更（段組 / 余白 / 用紙の向き）にはインラインアンカーがないため、トグルの隣にチップとして表示され、それぞれ独自の ✓/✕ を持ちます。原文（オリジナル）ビューはこれらを実際に元へ戻す（フォント、段数、ページ状態）ため、本物のビフォー/アフター比較ができます。

### Flatten-on-accept（アーキテクチャの中核）

**承認は物理的な操作であり、見た目だけの処理ではありません。** 承認時に、削除は DOM から取り除かれ、挿入はプレーンな（スタイル付き）コンテンツへとアンラップされ、すべてのリビジョン属性が除去されます。ラッパーは、*今回のラウンドの* Undo を機能させ続ける不可視の `data-undo` スパンへと縮退し、次の提案が到着した時点で一掃されます。

これが重要な理由 — 下流のすべてが**構造的に（by construction）** クリーンに保たれるからです:

- エージェントコンテキスト（`getText`/`getContext`）は *クリーンな射影*（削除を除外したもの）を読むため、次のターンの引用が新旧テキストにまたがることはありません — 破損が複利的に累積するループが発生しません。
- 文字数カウント、検索と置換、印刷、コピーはすべて実際の文書だけを参照します。
- レビュー途中のリロードも安全です: 承認の記録（`changeSetId::editId`）は永続化され、`applyEdit` は冪等（idempotent）で、メモリ内の Undo マップが失われた場合、却下は DOM レベルの復元にフォールバックします。

### バッチ処理の UX

プランがバッチを宣言している場合（「先做第一批…」＝「まず第 1 バッチを実行…」）、承認されたターンには **継続下一批 ›**（次のバッチへ続行）に加えて、**⚡自動続批**（自動バッチ継続）のオプトイントグル（永続化される）が表示されます。自動継続は、承認のたびに「下一批」（次のバッチ）を送信します — 直列に、各バッチが再アンカー・再レビューされ — 連続バッチ数は 5 回が上限です。

## Excel: 変更前状態（before-state）のリプレイ

提案時に、デスクトップアプリは変更対象となった各セルの**完全な変更前状態** — 値 *および数式*、塗りつぶし、フォント色、太字 — をキャプチャします。却下（または原文ビュー）は、その操作が触れた次元だけを正確にリプレイします: 値の編集を却下しても同じセルのスタイル編集は壊れず、数式は数式として復元され、ユーザーの塗りつぶしは維持されます。原文/改後のクイックトグルは項目ごとの決定を尊重します（却下済みの編集は、ビューを切り替えても復活しません）。

## レール（両フォーマット共通）

常時表示される **git スタイルの unified diff**: `@@ ref label` のハンクに、赤の − / 緑の + 行と `~` の書式行、項目ごとの承認/却下、進捗バー、Undo 付きの「已采纳 · N 处」（採用済み・N 件）。ハンクにホバーすると対応するインライン変更がハイライトされ、その逆も同様です。

## テレメトリ

項目ごとの各決定は、フォーマット × 変更種別（`text` / `style` / `value` / `structure` / `object`）をキーとする `localStorage['oa.telemetry']` のカウンターをインクリメントします。コンソールから `__otterTelemetry()` で読み取れます。カテゴリごとの採用率は、「エージェントは本当に熟練しているか？」を測る正解データ（ground truth）の指標であり、スコアが最も低いカテゴリが次のプレイブック/プロンプト改善のターゲットになります。
