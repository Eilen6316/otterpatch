# エージェントループ

モデルにできることのすべてと、それを誠実に保つための仕組み。

## ルーティング（3 つの出口、1 つのループ）

システムプロンプト（`ROUTING_PREAMBLE`、`packages/agent/src/prompts/agent-loop.ts`）が契約を固定する：モデルはすべてのターンを、ちょうど 1 つのツール呼び出しで終えなければならない —

- `answer_user` — 質問・コンサルティング用。ドキュメントには一切触れない
- `propose_changeset` — **唯一の**変更（ミューテーション）出口。まず計画し、編集は反映前にレビューされる
- `ask_user` — 意図が本当に曖昧で、推測のコストが高い場合に使う、ガイド付きの確認テーブル（各質問につき 2〜4 個の選択肢）

その間、モデルは最大 `STEP_LIMIT = 8` ループステップまで**読み取りツール**を呼び出せる。両方のモデルチャネル（`anthropic.ts`、`openai-compat.ts`）は、`sheet-tools.ts` / `doc-tools.ts` にある共有のプロバイダー非依存のツール定義の上で、同一のループを実装している。

## 読み取りツール（行動する前に知覚する）

| フォーマット | ツール | 目的 |
|---|---|---|
| Excel | `read_range` | 任意の A1 範囲の正確なセル値（サンプルから推測しない） |
| Excel | `aggregate` | `groupBy` / `where` 付きの列集計 — ピボット、合計、異常値統計 |
| Word | `read_blocks` | 段落範囲の全文（プロンプトのコンテキストは長い段落を切り詰めるため、引用は実際のテキストから取らなければならない） |
| Word | `find_text` | ブロック番号付きの全出現箇所 — 引用の一意性チェック |
| Word | `get_outline` | 見出しツリー + レベル飛ばしの診断 |
| Word | `get_style_usage` | スタイル・フォント・サイズ・配置の分布 — タイポグラフィ監査の素材 |
| any | `load_skill` | ドメインプレイブックの完全な指示を取得する（[skills.md](./skills.md) を参照） |

スナップショットはリクエストに同乗する（`ProposeRequest.sheet` / `.doc`）が、ツールからのみ見える — プロンプトには貼り付けられない。

## シャドー検証：propose → observe → repair

すべての `propose_changeset` は、diff になる前に検証される（検証器レジストリは `packages/runtime/src/runtime.ts` の `registerVerifier(format, make)`）：

- **Excel**（`buildGridVerifier`）— 再計算 + 範囲境界 + 重複ヒットのチェック
- **Word**（`buildDocVerifier`）— すべての引用がソーステキスト内に存在し、**かつ一意である**こと。空の編集と重複ヒットはフラグが立てられる
- **drawio**（`buildDrawioVerifier`）— トポロジーの整合性：`update/delete/move` の対象が存在すること。新しいエッジは既存ノードまたは同一提案内のノードに接続していなければならない（宙ぶらりんのエッジは禁止）。重複 id は禁止

失敗は同一ターン内で構造化レポートとしてモデルに返され、モデルは最大 `maxRepairs = 2` 回まで修復できる。

**最終セルフチェック**（`withFinalSelfCheck`）：*大きな*チェンジセット（編集 5 件以上）が構造検証を通過すると、モデルは「自分の仕事を全体としてレビューする」ラウンドをちょうど 1 回だけ得る — 完全性、衝突、より良いアプローチ — その後、再提出する（満足なら変更なしで）。小さなチェンジセットはこれをスキップする。

## プロンプトキャッシング

Anthropic チャネルはシステムプロンプトを、**安定プレフィックス**（ルーティング + 方言 + スキル — ターンをまたいで同一）と**揮発テール**（このターンのドキュメントスナップショット）に分割し、それぞれに `cache_control` ブレークポイントを付ける。結果：8 ステップのループの各ステップで、システムプロンプト全体がキャッシュにヒットする。ターンをまたいでも安定プレフィックスは引き続きヒットする。

## バッチ処理（直列、決して並列にしない）

長い出力はバッチに分割される：計画は「最初の N 項目」を宣言し、受け入れ後、ユーザーは **継続下一批**（次のバッチへ続行）をクリックできる — あるいは **⚡自動続批**（自動続行、オプトイン、永続化される）を有効にすると、各受け入れの後に自動で「下一批」（次のバッチ）が送信される。連続自動バッチは 5 回が上限。各バッチは、*現在の*ドキュメントにアンカーされた完全な propose → verify → review ラウンドである。

なぜバッチに並列サブエージェントを使わないのか？ すべてのアンカー（引用 / A1 参照）は 1 つのドキュメントリビジョンに対して解決される。バッチ A が反映された瞬間、バッチ B のアンカーは陳腐化する — サイレントなノーオペや、誤った位置への編集反映になる。並列性は**読み取り**（診断のファンアウト）には安全だが、書き込みは直列にアンカーされた単一のチェンジセットに収束しなければならない。将来サブエージェントを導入するなら、設計は次のとおり：並列リーダー群 → 1 つのライター → 1 つのチェンジセット → 1 つのレビュー。

## モデルが見る履歴と状態

`buildHistory` は過去の各ターンを 1 行に射影し、**正味の結果**を含める — 「ユーザーが N 項目を受け入れた」「ユーザーがこれらを差し戻した」— これにより、モデルは反映済みの変更を再提案したり、差し戻された変更の上に積み上げたりしない。承認状態はコンテキストのトリミングを生き延びる（削除されたターンはステータス要約を残す）。
