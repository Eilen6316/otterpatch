# Capability Bench — キャリブレーション記録

`test/expert-bench.mjs` は実モデルでエキスパートタスク集（単発 8 件 + マルチターン対話シナリオ 4 件）を実行し、二層で採点する：
客観的不変条件（応答タイプ / 必須ツール呼び出し / changeset の形状、アンカー含む）+ LLM-judge（1〜5 点）。
ラウンドごとの結果は `test/bench-results.jsonl` に追記される。実行方法はファイル先頭のコメント参照（key が無ければ自動 SKIP、CI セーフ）。

### 修正の全リスト（6 ラウンドの蓄積）
1. 確認のハード基準（prompt・ルーティング③）：目的が欠けたリクエストは必ず ask_user
2. STEP_LIMIT 8→12（loop）：エキスパートフローのステップ数予算
3. 続きバッチの宣言（prompt・ルーティング⑥）：続きバッチのターンではバッチを宣言し、書き込み済み項目を重複させない
4. グラフ種別のデフォルト（prompt・excel④）：カテゴリ比較はデフォルト bar、「割合」と言われていなければ pie を使わない
5. **readRange の型可視性（ツール層）**：テキスト数値を明示的にマーク——正確性に関わる重要情報はスキルのマッチングに頼らず、ツール出力自体に目を持たせる
6. **NUDGE_TOOLIFY（チャネル層）**：生テキストで締めたら一度だけツール化を促す——ルーティング契約「毎ターン 1 回のツール呼び出し」をコードで担保
7. bench のアンカーアサーション（テスト自身）：opsMust が anchors をカバー

### 方法論の蓄積
- 修正は三層に分けて切り込む：**プロンプト**（挙動の選好）→ **ツール出力**（情報の可視性）→ **チャネルコード**（契約の担保）；後ろの層ほど確実
- judge は綺麗な散文に騙されるが、客観的不変条件は騙されない——二層とも必要
- 単発スコアには分散がある（x-anomaly は 5/2/5/4 の間で変動）。2 ラウンド連続で再現して初めて真の弱点とみなす

### 残る観察項目
- mt-next-batch 3/5：続きバッチで前バッチと重複するセンシティブ項目がまだ時折発生——plan では宣言したが edits で完全には回避できていない
- w-gongwen 4/5：小標宋（中国公用文書の標題用書体）など近似フォントの適用境界
- 真の北極星は依然としてデスクトップ側の項目別受け入れ率テレメトリ（`__otterTelemetry()`）
